Архитектурные подходы

Как организовать внутренности антивируса?

A

Windows Service + GUI (рекомендую)

Два процесса: фоновый сервис (запускается при старте Windows, всегда работает) и GUI-приложение (показывает статус, управляет сервисом). Общаются через Named Pipes.

[Windows Service]
├── RealTimeGuard (FileSystemWatcher)
├── ScanEngine (YARA + ClamAV)
├── QuarantineManager
└── ← Named Pipes IPC →

[GUI App — Avalonia]
├── Dashboard
├── ScanView
└── Settings

Плюсы

  • Защита работает даже когда GUI закрыт
  • Так работают все настоящие антивирусы
  • Сервис стартует с Windows

Минусы

  • Сложнее: два проекта, IPC, права администратора для сервиса
B

Единый процесс с фоновыми потоками

Одно приложение: GUI + движок в одном процессе. Мониторинг запускается в фоне при старте приложения. Приложение добавляется в автозагрузку Windows.

[Единый процесс — Avalonia App]
├── UI Layer (MVVM)
├── BackgroundService (hosted)
│ ├── RealTimeGuard
│ ├── ScanEngine
│ └── QuarantineManager
└── Tray Icon

Плюсы

  • Проще в разработке
  • Один проект
  • Хорошо для старта, особенно с новым языком

Минусы

  • Защита не работает если пользователь закрыл приложение полностью
  • Иконка в трее обязательна
C

Гибрид: старт как B, рефакторинг в A

Начать с единого процесса (проще учиться C#), но сразу проектировать слои так, чтобы потом легко вынести движок в сервис. Модули изолированы через интерфейсы.

v1: Единый процесс с чистыми интерфейсами
→ IAntivirusEngine, IQuarantine, IRealTimeGuard

v2: Engine → Windows Service
GUI подключается к уже готовым интерфейсам

Плюсы

  • Легко начать
  • Правильная архитектура с первого дня
  • Нет технического долга при переходе

Минусы

  • Требует дисциплины при проектировании интерфейсов
